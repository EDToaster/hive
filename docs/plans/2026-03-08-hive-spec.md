# Hive: Agentic Swarm Coordinator

## Overview

Hive is a stateless tool that orchestrates a swarm of Claude Code agents for autonomous software development. It manages agent lifecycles, task state, message passing, worktree isolation, and a merge queue. All state lives in a `.hive/` directory in the repo root. Hive exposes an MCP interface for Claude Code agents and a CLI for humans.

## Architecture

Hive is not a daemon. It is a stateless binary that reads and writes `.hive/` state on every invocation. The coordinator (a Claude Code instance) drives all orchestration by calling hive repeatedly. A TUI mode provides read-only monitoring by watching `.hive/` for changes.

```
User
  |
  | writes spec, runs `hive start`
  v
Coordinator (Claude Code)
  | uses hive MCP tools to orchestrate
  |
  +-- Lead: Backend (Claude Code, has worktree)
  |     +-- Worker: implement endpoint
  |     +-- Worker: write migration
  |
  +-- Lead: Frontend (Claude Code, has worktree)
  |     +-- Worker: build form component
  |
  +-- Lead: Testing (Claude Code, has worktree, spun up mid-run)
        +-- Worker: integration tests
        +-- Worker: e2e tests
```

## Agent Hierarchy

### Coordinator
- Receives the initial spec from the user.
- Decomposes the spec into domain-level chunks and assigns them to leads.
- Dynamically spins up new leads mid-run if needed.
- Processes the merge queue: merges branches sequentially, runs tests, handles conflicts.
- Handles cross-domain issues (e.g., two leads modified the same file).
- Reports status to the user.
- Does NOT get a worktree. Does NOT read implementation code.

### Lead
- Receives a domain-scoped chunk of work from the coordinator.
- Gets its own worktree. Reads code to understand the domain.
- Decomposes its chunk into specific tasks and assigns them to workers.
- Reviews worker output (diffs) before approving.
- May send workers back with feedback.
- Submits approved work to the merge queue.
- Can suggest new tasks to the coordinator (e.g., cross-domain concerns).
- Can spawn its own workers.

### Worker
- Receives a specific, well-defined task from a lead.
- Gets its own worktree. Implements the task.
- May run bash commands (linters, tests, build tools) as part of its work.
- Submits completed work to its lead for review.
- Can suggest new tasks (routed to its lead), e.g., "found a bug while implementing this."
- Cannot spawn agents. Cannot submit to the merge queue directly.

## Roles and Permissions

| Capability           | Coordinator | Lead              | Worker                    |
|----------------------|-------------|-------------------|---------------------------|
| Create tasks         | yes         | yes (scoped)      | yes (suggestions to lead) |
| Spawn agents         | yes (leads) | yes (workers)     | no                        |
| Submit to merge queue| process only| yes (submit)      | no                        |
| Send messages        | to leads    | to workers + coord| to own lead               |
| Review code          | no          | yes               | no                        |
| Own a worktree       | no          | yes               | yes                       |

## Task System

### Task Properties
- `id`: unique identifier
- `title`: short description
- `description`: detailed spec, may include file paths and line numbers
- `status`: pending | active | blocked | review | approved | queued | merged | failed
- `urgency`: low | normal | high | critical
- `blocking`: optional list of task IDs this blocks
- `blocked_by`: optional list of task IDs blocking this
- `assigned_to`: agent ID (lead or worker)
- `created_by`: agent ID
- `parent_task`: optional, for subtask relationships
- `branch`: git branch name (set when work begins)
- `domain`: optional domain tag (backend, frontend, testing, etc.)

### Task Lifecycle

```
pending --> active --> review --> approved --> queued --> merged
              |          |
              v          v
           blocked     active (sent back with feedback)
              |
              v
           active (unblocked)

Any state --> failed
```

### Task Creation by Workers
Workers can create tasks with type `suggestion`. These are routed to the worker's lead, who decides whether to act on them, escalate to the coordinator, or dismiss. Suggestions carry the originating context (what the worker was doing when it found the issue).

## Message Passing

### Message Properties
- `id`: unique identifier
- `from`: agent ID
- `to`: agent ID
- `timestamp`: ISO 8601
- `type`: info | request | status | task-suggestion
- `body`: message content
- `refs`: optional list of task IDs for context

### Message Types
- **info**: context update, no response needed. Used for things like "user updated the spec" or "main branch has been updated, you may need to rebase."
- **request**: expects action from the recipient. Used for "rebase and resubmit" or "review this diff."
- **status**: agent reporting progress. Used for heartbeats, completion notices, stall reports.
- **task-suggestion**: worker proposing a new task. Routed to the worker's lead.

### Delivery
Messages are injected directly into the agent's Claude Code conversation as new conversation turns. The agent does not poll. Hive writes the message to `.hive/` and delivers it to the running Claude Code process.

## Merge Queue

The merge queue is an ordered list of approved branches waiting to be integrated into main. The coordinator processes it sequentially.

### Merge Process
1. Lead approves a worker's output and submits the branch to the merge queue.
2. Coordinator calls `hive_merge_next`.
3. Hive checks out the branch and attempts to merge into main.
4. Hive runs the test suite.
5. On success: mark task as `merged`, advance main, move to next item.
6. On conflict: coordinator attempts auto-resolution. If confidence is low, sends the lead back to rebase the branch against updated main and resubmit.
7. On test failure: sends the lead back with failure details.

## Worktree Management

- Leads and workers get worktrees via `git worktree add`.
- Setup takes ~60 seconds on large monorepos. This is acceptable.
- Worktrees are stored under a configurable path (default: `.hive/runs/<run-id>/worktrees/`).
- Each worktree gets its own branch: `hive/<run-id>/<agent-id>`.
- Cleaned up via `git worktree remove` when agent completes or run ends.
- The coordinator does NOT get a worktree.

## Agent Spawn Sequence

When `hive_spawn_agent` is called (by the coordinator or a lead), hive performs these steps:

1. **Create worktree**: `git worktree add .hive/runs/<run-id>/worktrees/<agent-id> -b hive/<run-id>/<agent-id>`
2. **Write hooks config**: `<worktree>/.claude/hooks.json` — configures Claude Code notification hooks to call `hive log-tool` on every tool invocation, with the agent ID and run ID baked in.
3. **Write MCP config**: `<worktree>/.claude/settings.json` — registers hive as a stdio MCP server: `hive mcp --run <run-id> --agent <agent-id>`.
4. **Write agent instructions**: `<worktree>/CLAUDE.local.md` — generated per-agent based on role. Uses `CLAUDE.local.md` (not `CLAUDE.md`) so that hive's instructions are additive to any existing project-level CLAUDE.md and are not checked into git. Contains:
   - Agent's role (coordinator, lead, worker)
   - Agent's parent (who it reports to)
   - Assigned task description
   - Behavioral constraints (what it can/cannot do)
   - Instructions for hive tool usage (how to report status, suggest tasks, submit work)
5. **Launch Claude Code**: start `claude` process with working directory set to the worktree.
6. **Register agent**: write agent metadata to `.hive/runs/<run-id>/agents/<agent-id>/` (pid, role, status, parent, worktree path, heartbeat).

The agent is born fully configured. It does not need any setup steps — it reads its CLAUDE.local.md (additive to any existing project CLAUDE.md), sees hive as an available MCP tool, and begins working.

### Generated CLAUDE.md Examples

**Lead:**
```markdown
You are a lead agent in a hive swarm.
Run ID: <run-id>
Agent ID: lead-backend
Role: lead
Parent: coordinator

## Your Assignment
<task description from coordinator>

## Responsibilities
- Decompose your assignment into specific worker tasks.
- Spawn workers via hive_spawn_agent for each task.
- Review worker output when they submit for review.
- Send workers back with feedback if changes are needed.
- Submit approved branches to the merge queue via hive_submit_to_queue.
- Report progress to the coordinator via hive_send_message.

## Constraints
- You may only spawn workers, not other leads.
- You may only send messages to your workers and the coordinator.
- Do not process the merge queue — the coordinator handles that.
```

**Worker:**
```markdown
You are a worker agent in a hive swarm.
Run ID: <run-id>
Agent ID: worker-003
Role: worker
Parent: lead-backend

## Your Task
<specific task description from lead>

## Responsibilities
- Implement the task in your worktree.
- Run relevant tests and linters to verify your work.
- When done, call hive_update_task to set status to "review".
- If you discover an unrelated bug or issue, call hive_create_task
  with urgency and a description. It will be routed to your lead.

## Constraints
- Do not spawn other agents.
- Do not submit to the merge queue directly.
- Do not send messages to agents other than your lead.
- Stay focused on your assigned task.
```

### Coordinator Spawn

The coordinator is special — it is spawned by `hive start`, not by another agent. It does not get a worktree. Its instructions are written to `CLAUDE.local.md` in the base repo (additive to existing project instructions, not checked into git). Its MCP config points to `hive mcp --run <run-id> --agent coordinator`.

## `.hive/` Directory Structure

```
.hive/
  config.yaml                  # hive configuration (human-readable)
  runs/
    <run-id>/
      spec.md                  # original spec (human-readable)
      tasks/
        task-001.json          # task state (human-readable)
        task-002.json
      agents/
        coordinator/
          pid
          role
          status               # running | done | failed | stalled
          heartbeat            # unix timestamp, updated periodically
        lead-backend/
          pid
          role
          status
          heartbeat
          worktree             # path to worktree
          parent               # coordinator
        worker-001/
          pid
          role
          status
          heartbeat
          worktree
          parent               # lead-backend
      messages/
        msg-0001.json
        msg-0002.json
      merge-queue.json         # ordered list of branches (human-readable)
      log.db                   # sqlite: tool usage, timing, events
      worktrees/               # git worktrees live here
```

## Observability and Tool Usage Logging

Hive tracks two categories of tool usage in `log.db`:

### 1. Hive API calls
Every invocation of a hive tool (MCP or CLI) is logged automatically. This includes who called it, what arguments were passed, and how long it took.

### 2. Agent tool calls
Every tool call made by any Claude Code instance (Read, Write, Edit, Bash, Grep, etc.) is captured via Claude Code notification hooks. When hive spawns an agent, it configures the agent's Claude Code instance with a hook that reports tool usage back to hive:

```
# Configured per-agent via Claude Code hooks
on tool_call: hive log-tool --run <run-id> --agent <agent-id> --tool <tool> --duration <ms> --status <success|error>
```

### log.db schema

```sql
CREATE TABLE tool_calls (
    id INTEGER PRIMARY KEY,
    run_id TEXT NOT NULL,
    agent_id TEXT NOT NULL,
    agent_role TEXT NOT NULL,       -- coordinator | lead | worker
    source TEXT NOT NULL,           -- hive | claude-code
    tool_name TEXT NOT NULL,        -- e.g. hive_spawn_agent, Read, Bash
    args_summary TEXT,              -- truncated/sanitized args for debugging
    status TEXT NOT NULL,           -- success | error
    duration_ms INTEGER,
    timestamp TEXT NOT NULL         -- ISO 8601
);

CREATE INDEX idx_tool_calls_agent ON tool_calls(run_id, agent_id);
CREATE INDEX idx_tool_calls_tool ON tool_calls(run_id, tool_name);
```

### TUI aggregations
The TUI dashboard queries `log.db` to display:
- Total tool calls per agent (who is busiest)
- Tool call breakdown by type (how many Reads vs Writes vs Bash calls)
- Error rates per agent
- Average tool call duration
- Timeline of activity (detect idle/stalled agents)
- Hive API call frequency (spawn rate, message volume)

## Interfaces

### MCP Tools (for Claude Code agents)

| Tool                 | Description                                              |
|----------------------|----------------------------------------------------------|
| `hive_spawn_agent`   | Create a new agent with a role, parent, and task         |
| `hive_send_message`  | Send a message to an agent                               |
| `hive_create_task`   | Create a task with urgency, blocking info, domain        |
| `hive_update_task`   | Update task status, assignment                           |
| `hive_list_tasks`    | Query tasks by status, assignee, domain                  |
| `hive_submit_to_queue` | Submit an approved branch to the merge queue           |
| `hive_merge_next`    | Process the next item in the merge queue                 |
| `hive_list_agents`   | List agents and their statuses                           |
| `hive_check_agents`  | Detect stalled/failed agents via heartbeat comparison    |
| `hive_log_tool`      | Record a tool call event (used by notification hooks)    |

### CLI (for humans)

| Command              | Description                                              |
|----------------------|----------------------------------------------------------|
| `hive init`          | Initialize `.hive/` in repo root                         |
| `hive start <spec>`  | Start a new run with a spec file                         |
| `hive status`        | Show current run status                                  |
| `hive agents`        | List agents and their health                             |
| `hive tasks`         | List tasks and statuses                                  |
| `hive messages`      | View message history                                     |
| `hive log-tool`      | Record a tool call event (called by agent hooks)         |
| `hive logs`          | Query the event log                                      |
| `hive tui`           | Launch the monitoring dashboard                          |

### TUI Dashboard

Read-only for now. Panels:
- Agent health (status, heartbeat, role, parent)
- Task board (grouped by status, filterable by domain)
- Merge queue (current item, queue depth)
- Tool usage stats (aggregate from log.db)
- Message feed (live tail of messages)

Reads from `.hive/` files and `log.db`. Uses file watching (`notify`) for live updates.
